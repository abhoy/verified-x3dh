# X3DH Verification Setup

This crate already contains:

- the Rust X3DH verification-friendly core
- the hax-facing annotations
- the F* extraction output directory layout
- a verification runner: `verify_x3dh`
- a setup wrapper: `install_x3dh_verification.sh`

The wrapper is meant to handle local configuration and then invoke the existing `cargo hax into fstar` plus F* verification flow.

## Repository Structure

This setup assumes only the local crate structure:

```text
verified_x3dh/
  src/
  Cargo.toml
  verify_x3dh
  install_x3dh_verification.sh
  INSTALL.md
```

The `hax` toolchain is treated as an external prerequisite. It does not need to live next to `verified_x3dh`, but its installation root must be known.

## Required hax-side F* directories

For this crate's manual F* verification flow, the configured `HAX_ROOT` must contain:

- `$HAX_ROOT/proof-libs/fstar/core`
- `$HAX_ROOT/proof-libs/fstar/rust_primitives`
- `$HAX_ROOT/hax-lib/proofs/fstar/extraction`

These are the include roots used by `verify_x3dh` when invoking `fstar.exe`.

## Prerequisites

Install the toolchain first.

For hax, the official manual says the manual installation prerequisites include:

- `opam`   (Source: https://opam.ocaml.org/doc/Install.html)
- `rustup` (Source: https://doc.rust-lang.org/nightly/cargo/getting-started/installation.html)
- `nodejs`
- `jq`
- `hax` (Source: https://hax.cryspen.com/manual/)
- `F*` (Source: https://github.com/FStarLang/FStar/blob/master/INSTALL.md)


## Files

- `verify_x3dh`
  Runs extraction and then verifies the extracted F* files.
- `install_x3dh_verification.sh`
  Generates local config, checks prerequisites, prepares the repo, and launches verification.
- `.x3dh_verify.env`
  Local machine-specific config file generated by the installer.

## Configure

From the repository root:

```bash
cd verified_x3dh
chmod +x verify_x3dh install_x3dh_verification.sh
./install_x3dh_verification.sh configure --hax-root /absolute/path/to/hax
```

If needed, set a specific opam switch or F* binary:

```bash
./install_x3dh_verification.sh configure \
  --hax-root /path/to/hax \
  --opam-switch 5.1.1 \
  --fstar-bin fstar.exe
```

This writes `.x3dh_verify.env` in `verified_x3dh/`.

If `HAX_ROOT` is already exported in your shell, you can reuse it:

```bash
export HAX_ROOT=/path/to/hax
./install_x3dh_verification.sh configure
```

`configure` now requires a valid `hax` root either through `--hax-root` or `HAX_ROOT`. The scripts no longer assume any sibling repository layout.

## Check Prerequisites

```bash
./install_x3dh_verification.sh check
```

This checks:

- `cargo`
- `cargo hax`
- `opam` when enabled
- the configured `HAX_ROOT`
- `$HAX_ROOT/proof-libs/fstar/core`
- `$HAX_ROOT/proof-libs/fstar/rust_primitives`
- `$HAX_ROOT/hax-lib/proofs/fstar/extraction`
- the configured F* binary

## Local Setup / Install

Prepare the extraction directory:

```bash
./install_x3dh_verification.sh setup
```

Run the local crate build/tests before verification:

```bash
./install_x3dh_verification.sh install
```

## Smoke Test

Before running hax/F* extraction, you can validate the executable Rust model directly with:

```bash
cargo test
```

You can also run the abstract handshake demo:

```bash
cargo run --example demo_core_handshake
```

This is separate from F* verification:

- `cargo test` checks the Rust executable model and regression tests
- `cargo run --example demo_core_handshake` runs a deterministic demo of the abstract X3DH core/state model
- F* verification checks extracted proof-oriented artifacts

## Run Verification

Full extraction plus full F* verification:

```bash
./install_x3dh_verification.sh run all verify-all
```

Verify already-generated F* files only:

```bash
./install_x3dh_verification.sh run verify-only verify-properties
```

Run only part of the extraction:

```bash
./install_x3dh_verification.sh run kdf verify-kdf
./install_x3dh_verification.sh run transcript verify-transcript
./install_x3dh_verification.sh run handshake verify-handshake
```

Run a partial hax extraction target:

```bash
./install_x3dh_verification.sh run partial verify-kdf '+verified_x3dh::kdf::x3dh_kdf'
```

The generic hax partial extraction form is:

```bash
cargo hax into -i '-** +your_crate::some_module::my_function' fstar
```

The hax quick start also notes:

- dependencies of the selected item are extracted automatically
- `+!` can be used instead of `+` when dependency inclusion is not wanted
- unsupported items can sometimes be excluded with `-your_crate::path::item`

Source: https://hax.cryspen.com/manual/fstar/quick_start/

Run the whole flow in one command:

```bash
./install_x3dh_verification.sh all all verify-all
```

## Direct Use Of The Runner

You can still invoke the underlying runner directly:

```bash
./verify_x3dh all verify-all
./verify_x3dh verify-only verify-properties
```

`verify_x3dh` now reads configuration from:

```text
verified_x3dh/.x3dh_verify.env
```

or from the environment variable:

```text
X3DH_VERIFY_CONFIG
```

## Notes

- F* verification will fail if `fstar.exe` is not available in the active environment or configured path.
- The extracted `.fst` files under `proofs/fstar/extraction/` are generated artifacts and may need to be refreshed after Rust changes.
- The script can load both the repo virtualenv and the opam environment, but both are configurable.
- The hax F* quick start recommends a two-step Makefile-based verification flow after extraction:
  `OTHERFLAGS="--lax" make` first, then `make`.
- In this crate, `verify_x3dh` currently performs direct `fstar.exe` verification rather than invoking the Makefile, but the Makefile remains available under `proofs/fstar/extraction/` if you want to follow the generic hax quick-start workflow.
